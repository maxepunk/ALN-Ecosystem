Date: 2026-06-09

- **intentional** — hardware/platform constraint, rationale documented; spec records it as an allowed deviation

## 3. Draft Player-Scanner Role Spec

### Role (engine-level)

A **player scanner** is an intel-gathering device. It:

1. **Accepts a physical token** (NFC/RFID tag or QR code) and extracts a token identifier from its NDEF Text record / QR payload.
2. **Canonicalizes the tokenId** to `^[a-z0-9_]+$` (lowercase; strip separators) before any lookup. *(Web-conformant today; ESP32 fix pending — F-PARITY-01.)*
3. **Validates against the local token database** (tokens.json). Unknown tokenIds are a local failure: show feedback, do NOT transmit. The orchestrator must only ever receive real game tokenIds.
4. **Displays the token's memory content locally** — image and/or audio — immediately and regardless of connectivity. This is the core function and MUST work fully offline.
5. **For video tokens**, shows a distinct "this triggers the big screen" treatment instead of/alongside local content *(exact UX pending owner decision — F-PARITY-05)*; actual playback is the orchestrator's job, triggered by the scan report.
6. **Reports every known-token scan** to the orchestrator (`POST /api/scan`, fire-and-forget: `{tokenId, teamId?, deviceId, deviceType, timestamp}`) when reachable; otherwise **queues locally** (cap 100, FIFO overflow dropping oldest, order preserved) and **replays in batches of 10** (`POST /api/scan/batch`) with a globally-unique batchId when connectivity returns. Scan reporting must NEVER delay or block local content display.
7. **Allows duplicate scans** — players may re-view any memory; no duplicate rejection at any layer.
8. **Identifies itself stably** — persistent deviceId, correct deviceType (`player` | `esp32`), and a 10-second `/health?deviceId=...` heartbeat while in a connected deployment.
9. **Never scores, never commits teams.** `teamId` is optional decoration (currently both implementations default it to `001` — see F-PARITY-10).

### Documented per-implementation deviations (intentional)

| Concern | Web PWA | ESP32 | Rationale |
|---|---|---|---|
| Input medium | Camera QR + Web-NFC + manual entry + URL param | MFRC522 RFID (NDEF Text only) | Hardware |
| Media formats | Any browser-supported image/audio, paths from tokens.json | 24-bit BMP + WAV, paths CONSTRUCTED from tokenId (`/assets/images/{id}.bmp`, `/assets/audio/{id}.wav`) — tokens.json media fields ignored except `video` | SD/flash constraints; deterministic asset sync |
| Dismissal | Single tap / Esc | Double-tap ("Escape Memory") | Resistive-touch accidental-tap rejection |
| Token/asset freshness | Build-time submodule + SW cache (manual cache-bump refresh) | Boot-time wireless sync (tokens via `/api/tokens`, assets via SHA-1 manifest diff) | Deployment model |
| Deployment stance | Dual-mode, path-detected: standalone (never connects, no queue) vs networked | Single mode: offline-resilient, always sync-intending (queues forever, 10s reconnect probe) | Web has a no-backend product (GitHub Pages); the hardware scanner exists to feed the orchestrator |
| Connection state model | Binary connected/offline, passive header pill | 3-state (no WiFi / WiFi only / orchestrator), on-demand status screen | Platform diagnostics depth vs screen budget |
| Queue persistence | localStorage JSON array | SD `/queue.jsonl` + corruption guard + stream-based rebuild | Storage media |
| Failure feedback | Generic 3s toast | Differentiated non-blocking SCAN_FAILED screens (COMM/READ/UNKNOWN), re-scan not blocked | Hardware failure modes are richer |

### Known non-conformances to this spec (tracked above)

ESP32: raw-text DB lookup (F-01), batchId reuse (F-02), 409-drop of pre-session scans (F-03), blocking send-before-display (F-06). Web: 409 re-queue (F-03), unstable deviceId (F-04), cache-first token staleness (F-07), UID-as-tokenId NFC fallback (F-13).

## 5. Open Questions for Owner

1. **409 policy (F-03, P1):** When the backend answers 409, should the player scanner (a) queue-and-retry `SESSION_NOT_FOUND` scans (preserves pre-show scans) and (b) accept video-busy as final? Today ESP32 drops both; web re-queues both. Requires both clients to inspect the response body to distinguish.
2. **Video-token player experience (F-05):** Is the role behavior "dramatic alert + normal memory display" (web) or "processing modal, content suppressed" (ESP32)? And should the offline/queued case show a different ("will play later"?) treatment? Is `processingImage` a live schema field or dead weight?
3. **Local memory log (F-08):** Is the collection/"New vs Revisited" feature part of the role? If yes → ESP32 needs an SD scan log + revisited indicator; if no → strip web's vestigial `viewCollection()` and the Set, or keep the Set purely for the revisited badge.
4. **teamId on player scanners (F-10):** Contract says players haven't committed to teams. Should player scanners stop sending teamId entirely (cleanest), or should TEAM_ID/3-digit (ESP32) and the dead web mechanism be aligned to one definition?
5. **Batch per-item failures (F-09):** Is fire-and-forget acceptance of partial batch failure the intended contract (server-side audit only), or should devices retain/log failed items?
6. **ESP32 passive connectivity indicator (F-14):** Should the ReadyScreen carry a small online/offline/queue-depth glyph so staff can glance-check without tapping?
7. **tokenId canonical form (F-01):** Confirm `^[a-z0-9_]+$` lowercase as the role-wide canonical tokenId (it's already the asset-endpoint contract), so the ESP32 lookup normalization can be implemented without ambiguity.
8. **Web standalone token freshness (F-07):** Is "tokens frozen at deploy time" acceptable for the GitHub Pages product, or should the SW go network-first for tokens.json?
